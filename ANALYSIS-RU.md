# Анализ результатов: SHA-256 PoW benchmark

Документ разбирает референсные прогоны из `results/bench-oniguruma-*.json`
и интерпретирует их через минимальный комплект формул из теории
вероятностей. Аудитория — инженеры, которым из прикладной математики
нужны только те утверждения, которые используются дальше по тексту.

## 1. Постановка измерения

Задача solver'а — найти такой целочисленный nonce, чтобы первые `N`
hex-разрядов `sha256(token || ascii_decimal(nonce))` были нулями.
Каждый перебранный nonce — независимое испытание Бернулли. Параметр
сложности `N` управляет одной величиной — вероятностью успеха
`p = 16^{-N}`. Всё, что мы измеряем, — следствие из этого факта плюс
скорости хэширования `r` (hashes/sec).

Headline-метрика бенчмарка: максимальный `N`, при котором медианное
время решения укладывается в заданный бюджет (60 секунд в reference-прогонах).
Эквивалентная формулировка — наибольший `N`, при котором вероятность
`P(T ≤ target)` не ниже 1/2; ноутбук рядом печатает эту вероятность по
каждой ячейке с биномиальным интервалом Уилсона, чтобы у близких к границе
прогонов не возникало иллюзии точности.

## 2. Математические вводные

### 2.1 Распределение числа попыток

Число попыток до первого успеха — геометрическая величина `A ~ Geom(p)`.
Для уровня сложности `N`:

```
P(A = k)  = (1 - p)^{k - 1} · p
E[A]      = 1/p     = 16^N
Var[A]    = (1-p)/p^2 ≈ 16^{2N}
σ(A)/E[A] = sqrt(1 - p) ≈ 1   (при p << 1)
```

Отсюда первый практический факт: разброс `A` имеет тот же порядок,
что и его среднее. Удвоение или утроение времени решения относительно
ожидаемого — норма, а не аномалия.

### 2.2 Переход к wall-clock

Время решения `T = A / r`. При фиксированной `r` распределение `T` —
дискретное (`A` целочисленное), но при `p << 1` оно близко к
непрерывному экспоненциальному с интенсивностью `λ = r / 16^N`:

```
T ~ Exp(λ),  λ = r / 16^N
F(t) = 1 - exp(-λ t)
```

Относительная ошибка приближения — `O(p) = O(16^{-N})`, поэтому при
`N ≥ 4` разница между геометрическим и экспоненциальным распределением
исчезает на фоне выборочного шума.

### 2.3 Почему экспонента, а не нормальное

Интуиция «много независимых попыток → колокол» опирается на ЦПТ:
сумма большого числа независимых вкладов асимптотически нормальна.
В нашей задаче усреднения нет. Мы наблюдаем момент *первого* успеха
в потоке независимых бернуллиевых испытаний — это first-passage,
не сумма.

Эквивалентно: успехи приходят как пуассоновский процесс с
интенсивностью `λ`, и время до первого события пуассоновского процесса
— это по определению `Exp(λ)`. Никакая колоколообразная форма из этой
постановки получиться не может.

Memorylessness объясняет, почему дисперсия не «сглаживается с
опытом»: после `s` секунд бесплодного поиска остаточное время
по-прежнему `Exp(λ)`. Каждая попытка независима, накопленной
информации нет. `16^N` попыток в среднем — это не `16^N` слагаемых
в дисперсии, а `16^N` способов *не* успеть до сих пор; вся
стохастика задачи приходится на единственный успешный исход.

Норма всё-таки появляется в наших же данных, но в задачах с
усреднением:

- Калибровочный hashrate распределён примерно нормально вокруг
  ~1.2 GH/s — за две секунды складываются миллиарды мелких
  независимых интервалов между хэшами, и ЦПТ работает буквально.
- Время найти `K` решений подряд при больших `K` есть сумма
  `K·Exp(λ) = Gamma(K, λ)`, асимптотически нормальное с
  `CV ≈ 1/√K`. Если бы headline требовал «найти 100 решений
  с четырьмя нулями» вместо «одно решение с восемью»,
  эмпирические времена образовали бы колокол с `CV ≈ 0.1`,
  а не экспоненту с `CV = 1`. Та же физика, другой момент
  остановки.

Короткий критерий, какое распределение ожидать:

- сумма/среднее `n` независимых компонентов → нормальное (ЦПТ);
- время до первого редкого события → экспоненциальное (Пуассон);
- время до `K`-го события при больших `K` → опять нормальное,
  через гамму.

PoW попадает в средний случай: ровно одно редкое событие, и в нём
вся стохастика задачи. Увеличение `runs` не меняет формы
распределения — оно сужает доверительный интервал на оценку `λ`,
но не превращает экспоненту в колокол.

### 2.4 Характеристики Exp(λ), которые нужны для интерпретации

```
E[T]               = 1/λ           = 16^N / r
median(T)          = ln 2 / λ      ≈ 0.693 · E[T]
p95(T)             = -ln 0.05 / λ  ≈ 2.996 · E[T]
p99(T)             = -ln 0.01 / λ  ≈ 4.605 · E[T]
σ(T)               = 1/λ           = E[T]
CV = σ(T)/E[T]     = 1
p95(T)/median(T)   = ln 20 / ln 2  ≈ 4.32
p99(T)/median(T)   = ln 100 / ln 2 ≈ 6.64
```

Из этих соотношений следуют две проверки качества измерения:
коэффициент вариации эмпирических `T` должен быть около 1, отношение
p95 к медиане — около 4.32. Их выполнение — индикатор того, что
solver не вносит ничего, кроме линейного масштабирования по `r`.

### 2.5 Почему headline-метрика на медиане

Среднее `Exp(λ)` имеет дисперсию того же порядка, что само среднее.
Оценка среднего по `n` сэмплам сходится медленно: стандартная ошибка
падает как `1/sqrt(n)`, а в reference-прогонах `n` доходит до 1 на
большом `N` из-за per-N бюджета. Медиана же — порядковая статистика;
её оценка устойчива к одиночным выбросам и не требует допущения о
конечных моментах. Headline «макс `N`, где median(T) ≤ 60 с»
интерпретируется буквально: «успеть с вероятностью не меньше 1/2».

## 5. Соответствие модели Exp(λ)

CV для всех ячеек с `n ≥ 20` лежит в диапазоне 0,82–1,12 при
теоретическом значении 1. Отношение p95/median для тех же ячеек —
3,35–5,39 при теоретическом 4,32. Систематических отклонений нет:
крайние значения объясняются конечным размером выборки (для
`n = 20` стандартная ошибка p95 уже составляет десятки процентов от
самой оценки).

Случаи `n < 20` (shani N=8 имеет 20, shani N=9 имеет 1, python N=6
имеет 13, python N=7 имеет 5) лежат в зоне шумных оценок. По ним
осмысленно опираться только на медиану. Все остальные статистики
там диагностические, а не определяющие.

Гистограммы в ноутбуке (cell 13) визуально подтверждают
экспоненциальную форму для shani N=6 и заметный тяжёлый правый хвост
для python N=6. Тяжёлый хвост вне модели Exp(λ) объясняется не
вероятностью попадания в нулевой префикс, а внешним джиттером: GC
паузы CPython, шум планировщика на однопоточной нагрузке, переключения
контекста. Источник джиттера локализуется отдельным профилированием —
в текущем прогоне такие измерения не сняты.

## 6. Headline-метрика и связь с hashrate

shani достигает `max N = 8`, python — `max N = 6`. Разница в 2 hex-
разряда соответствует множителю `16² = 256` в числе ожидаемых попыток.
Соотношение эффективных hashrate: `397 / 1,33 ≈ 298`. Эти две цифры
согласованы: `log_16(298) ≈ 2.04`, и при равных бюджетах прирост
hashrate в `K` раз ровно конвертируется в прирост `max N` на
`log_16(K)`. Округление вниз до целого — единственный источник
небольшой нелинейности.

На фиксированном хосте `max N` — логарифмическая по hashrate
величина. Удвоить hashrate — это +0,25 hex-разряда; чтобы добавить
один целый разряд, требуется ускорение в 16 раз. Этим объясняется,
почему на одной машине SHA-NI с 32 потоками даёт всего +2 разряда
против однопоточного Python, несмотря на 700× разницу в калибровке.

## 7. Калибровка против эффективной hashrate

shani в калибровке выдаёт 1 207 MH/s parallel, в solve — 397–406
MH/s. Соотношение 0,33, то есть **67% времени параллельный solver
тратит не на хэширование**. python в калибровке выдаёт 1 730 MH/s,
в solve — 1 330 MH/s. Соотношение 0,77, налог однопоточного solver'а
на координацию — 23%.

В ноутбуке эти два числа теперь явно разнесены. Основной график рисует
две теоретические линии на каждую реализацию: пунктиром — нижнюю границу
`ln 2 · 16^N / r_calib` (недостижимый предел чистой `sha256` петли) и
штриховой — `ln 2 · 16^N / r_eff` с пулом `Σ attempts / Σ elapsed` по
всем прогонам данной реализации. Эмпирическая медиана с bootstrap-CI
ложится близко к штриховой линии и далеко выше пунктирной; вертикальный
зазор между ними — тот самый налог на координацию. Диагностический блок
в отдельной ячейке нормирует те же величины: `mean A / 16^N` показывает,
работает ли вероятностная модель (ожидаемо около 1), `median T /
(ln 2 · 16^N / r_calib)` — мультипликатор overhead'а к нижней границе,
`r_eff / r_calib` — то же отношение, выраженное через rate. Все три
графика разделяют «модель неправильная» и «solver тяжелее модели».

python однопоточный, и overhead над голым `hashlib.sha256` сводится
к проверке байтового префикса digest и инкременту nonce — обе операции
линейны по числу попыток, без скрытых синхронизаций. shani параллельный,
и его 67% потерь распадаются на несколько составляющих, которые в
текущем прогоне не разделены:

- запуск `rayon::scope` на каждом solve (миллисекундный масштаб);
- atomic-poll shared флага `found` каждые 4096 хэшей в 32 потоках;
- `Arc<Mutex<Option<...>>>` для записи решения и пробуждение чтения
  завершающих потоков;
- незавершённый последний батч у 31 «лишних» потоков после того,
  как один поток нашёл решение;
- аллокация `token.to_vec()` для каждого потока при spawn.

Декомпозиция этого 67% — задача профилирования; в данных её нет.
До профайлера цифра 67% — единственная агрегированная оценка налога
на параллелизм для текущего solver'а.

## 8. Per-N бюджет и его искажающий эффект

Бенчмарк ограничивает суммарное wall-clock на один `N` величиной
`max(3 × target, 30 s)` = 180 с в reference. Это объясняет, почему
число прогонов падает к границе допустимого `N`: shani N=8 успевает
20 из 30 прогонов, shani N=9 — один. Аналогично python обрывается на
N=6 (13 прогонов) и N=7 (5 прогонов).

Ограничение делает медиану устойчивой, но статистики хвоста — нет.
p95 при `n = 5` оценивается на максимальном значении выборки, что
эквивалентно «p80 ± шум». Headline остаётся честным, диагностические
колонки p95/p99 рядом с границей применимы только как «есть/нет
сигнала о тяжёлом хвосте».

## 9. Сравнение реализаций на пересечении

На `N = 6` доступны обе реализации одновременно. Медианы: shani
0,034 с, python 8,22 с. Прямое ускорение shani относительно python
на одном `N` — 242×. На том же `N` ECDF в ноутбуке (cell 11) для
shani почти ложится на теоретическую `1 - exp(-λt)`; python заметно
ниже теоретической кривой в области малых `t`. Это означает: у
python эмпирическое медленнее предсказанного по калибровке, что
согласуется с тем же 23% overhead, который мы видели в разделе 7.

## 10. Что измерение не покрывает

Один хост — нет сравнения между машинами и оценки разброса между
машинами одного класса. Reference-прогон не фиксирует CPU governor,
turbo-state и микрокод программно; снимок `env-*.txt` для этих
файлов в репозитории отсутствует, что мешает воспроизводимости на
другой стороне. Калибровка измеряет «голый» цикл sha256 — она не
содержит координации, поэтому переоценивает rate для параллельного
solve на коэффициент 3. Корректно интерпретировать `calibration.parallel_hps`
как нижнюю границу теоретического минимума `median(T) = ln 2 · 16^N / r`;
эмпирические медианы лежат на 3 раза выше этой границы для shani и
на 30% — для python.

Список открытых вопросов короткий: декомпозиция 67% overhead'а
параллельного solver'а и природа тяжёлого правого хвоста у python.
Оба решаются стандартным профилированием — `perf` для Rust,
`py-spy` или `cProfile` для Python — но в текущих данных их нет.

## 11. Что делать дальше, если хочется точнее

Для разделения координационных затрат: добавить в solver счётчики
`hashes_per_batch`, `flag_polls`, `rayon_scope_setup_ns` и записывать
их в JSON. Это даст ту же декомпозицию, что обещают пять пунктов в
разделе 7, без отдельного профайлера.

Для устойчивых хвостовых оценок: увеличить `runs` по N, где сейчас
`n < 20`. Точечные p95/σ в этих ячейках ноутбук уже маскирует
(`NaN`), на основной картинке усы интерпретируются как сырой spread
наблюдений p2.5–p97.5, а не как доверительный интервал; для медианы
показан bootstrap-CI, который честно сужается с ростом `runs`.
Дополнительный шаг — bootstrap-CI на p95 с порогом по относительной
ширине — остаётся открытым, но не блокирует headline.

Для воспроизводимости между машинами: коммитить пару
`bench-*.json` + `env-*.txt` атомарно; добавить в env-снимок
`/sys/devices/system/cpu/intel_pstate/no_turbo` (Linux) или
эквивалент на macOS. Тогда числа с двух машин сравниваются прямо.
